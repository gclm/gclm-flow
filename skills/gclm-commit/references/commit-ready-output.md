# Commit Ready Output

用于在验证通过后，主动给出可提交状态和候选 commit message，减少用户反复口述“帮我 commit”。

## 何时查看

- 相关测试、构建、lint 或专项验证已经成功
- 当前 Git 工作区仍有待提交改动
- 用户没有明确说“不提交”，但提交已经是自然下一步

## 推荐输出

```markdown
## Commit Ready
- 验证：`<已运行命令>` 已通过 / 已完成
- 改动范围：`<模块或文件组>`
- 拆分建议：单提交 / 建议拆成两次提交
- 候选提交：
  1. `<type>(<scope>): <中文描述>`
  2. `<type>(<scope>): <中文描述>`
- 下一步：如果你要我直接提交，我就按上面的候选信息执行。
```

## 重点做法

- 先说明验证证据，再给候选 message
- 候选标题只描述当前改动，不预支后续工作
- 如果暂存区混入无关文件，要先提示拆分，不要直接给“看起来很完整”的 message
- 当你判断提交是自然下一步时，主动给出 `Commit Ready`，不要等用户再次口述“commit”

## 注意事项

- `Commit Ready` 是建议，不等于擅自提交
- 没有验证证据时，不要假装进入 commit-ready 状态
