# Migration Safety Checklist

用于 schema、数据回填、键名迁移、索引变更的安全检查。

## 何时查看

- 变更会改表结构、索引、字段含义或数据格式
- 需要上线迁移脚本、回填脚本或双写逻辑
- 需要把现有配置/文件存储逐步迁移到数据库

## 检查清单

- 是否影响旧代码读取或写入
- 是否需要兼容窗口、双写、灰度发布
- 是否会长时间锁表、阻塞写入、影响启动
- 是否有回填脚本和失败重试方案
- 回滚时旧版本是否还能正确读取新数据
- 部署文档、默认值、配置模板是否同步修改

## 重点做法

### 替换持久化后端时，先收窄迁移面

- 例如从本地配置存储迁到 SQLite，第一阶段只迁移目标持久化层；YAML、日志等非目标文件先保持不变
- 每完成一组迁移就跑定向测试和构建，避免一次性重构到位后无法定位回归
- 迁移前先确认旧数据如何导入、新数据如何回滚、旧代码是否仍能安全读取

### 生成 SQL / 初始化数据时，拆分 fetch、transform、import 三步

- 先验证输入来源、输出路径、模板文件是否存在，再生成 SQL
- 生成物应尽量可重跑、可审查，不要把“抓取远端数据 + 拼 SQL + 导入”揉成一个黑盒脚本
- 导入失败时要能定位是源数据、生成逻辑还是执行环境的问题

## 注意事项

- 迁移问题优先看“能否平滑上线和回滚”
- 没有回滚思路的迁移默认按高风险处理
- 当前证据更偏 migration / bootstrap 安全，query 优化仍需要更多真实样本再继续沉淀
