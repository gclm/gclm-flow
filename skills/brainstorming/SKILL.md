# 方案设计

先收敛目标与边界，再进入实现。

> **HARD-GATE**: 设计未经用户确认前，不得写任何代码、修改任何文件、或调用任何实现 skill。无论任务看起来多简单，都不例外。用户确认设计后，此门解除，可调用 `writing-plans` 或 `dispatching-parallel-agents`。

## 何时使用

- 新功能、行为变更、工作流编排、较大配置改造
- 需要在 2 个以上方案之间做取舍
- 需求不完整、成功标准不清晰、风险较高

## 问题澄清（复杂或模糊问题优先执行）

回答复杂或模糊的问题前，先执行以下步骤：
1. 指出问题中隐含的假设。
2. 说明哪些关键信息缺失会显著改变答案。
3. 说明处理此类问题时最常见的一个错误。

然后向用户提一个能让最终答案真正产生价值的问题，等用户回答后再给出最终输出。

适用于：架构决策、方案选型、需求分析、或任何结论会因上下文显著变化的场景。
简单明确的执行类任务（如修 bug、写代码）跳过此步骤。

## 工作流

1. 只读探索：代码、文档、约束、现状。
2. 澄清目标：一次只问一个问题，优先选择题而非开放题；聚焦范围、非目标、验收标准、风险。
3. 给出 2-3 个方案，写清 trade-off 和推荐项。
4. 输出设计结论：架构、数据流、错误处理、验证方式；逐段呈现，每段确认后再继续。
5. 用户确认设计后，将结论写入 `docs/specs/YYYY-MM-DD-<topic>.md` 并提交 git。
6. 判断任务规模，选择执行路径：
   - **简单任务**（单模块、步骤清晰、无并行价值）→ 调用 `writing-plans` 生成串行计划
   - **复杂任务**（多模块、可并行调研或实现、需独立验收）→ 调用 `dispatching-parallel-agents`，用 `spawn_team` 启动团队

## 产出要求

- 结论先行，不写散乱脑暴
- 明确推荐方案和不选其他方案的原因
- 如果要改代码或配置，必须给出受影响文件/模块范围
- spec 文档只记录已确认的设计结论，不记录过程讨论
