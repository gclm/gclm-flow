# Handling Review Feedback

当 review comment 已经出现时，用这份清单处理，不再切换到另一个独立 skill。

## 处理顺序

- 先判断评论是否真的指出了问题，而不是只是一种偏好
- 检查评论是否和当前需求、平台约束、兼容性要求冲突
- 对模糊评论先澄清，不要只修你猜到的一半
- 外部 reviewer 的建议如果和代码库现实不符，要直接指出原因

## 常见输出

- 已修复：`已修复，变更位于 ...，原因是 ...`
- 需澄清：`我理解 X，但 Y 还不明确，需要确认 ...`
- 技术反驳：`当前实现保留此逻辑是因为 ...；若改成 reviewer 建议，会影响 ...`

## 红线

- 不要在没验证前说“完全正确，我马上改”
- 不要因为 reviewer 强势就跳过本地验证
- 不要批量改完再统一看结果，先处理阻塞项
