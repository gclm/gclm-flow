# Removal Plan

当变更涉及删除、下线、弃用、迁移时，额外检查以下内容。

## 适用场景

- 删除旧模块、接口、命令、页面、配置项
- 用新实现替换旧实现
- 数据结构、存储格式、配置键名迁移
- 清理 feature flag、兼容分支、临时补丁

## 检查清单

- 依赖分析：调用方、脚本、定时任务、CI、文档、监控、告警是否仍引用旧对象
- 兼容策略：是否需要灰度、双写、兼容窗口、弃用提示
- 数据安全：删除是否会造成数据丢失、历史记录不可读、回填失败
- 配置迁移：环境变量、默认值、部署模板、密钥引用是否同步调整
- 回滚路径：如果上线后出问题，能否快速恢复旧路径
- 测试覆盖：是否覆盖删除后的主路径、缺失依赖、回滚/降级场景
- 文档同步：README、运维文档、接口说明、迁移公告是否同步

## 高风险信号

以下情况至少应视为 `P1`，严重时 `P0`：

- 只删代码，不查调用方
- 删除兼容逻辑，但仍有旧客户端或旧数据存在
- 改了配置键名或 schema，没有迁移/回滚方案
- 删除监控、告警、审计日志，导致故障不可观测
- 依赖人工步骤，但变更说明没有明确记录

## 审查输出建议

发现删除类风险时，优先说明：

- 被删除对象是什么
- 哪些调用方或运行路径会受影响
- 缺少哪一项迁移/兼容/回滚措施
- 最小补救措施是什么
