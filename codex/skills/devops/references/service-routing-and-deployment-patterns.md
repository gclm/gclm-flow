# Service Routing And Deployment Patterns

用于代理分组、部署选址和 fork 场景下的发布脚本治理。

## 何时查看

- 需要给 GitHub、Cloudflare、Vercel、Telegram、微软、苹果等服务做规则分组
- 在 PVE / 内网环境 与 公网云主机之间选择部署位置
- fork 项目后要保留上游部署脚本，同时做自定义镜像/仓库注入

## 重点做法

### 服务路由按“能力桶”分组，不要为每个域名单独长期堆规则

- 软件开发相关服务优先聚合到一个开发组，例如 GitHub、Cloudflare、Vercel
- Telegram、Microsoft、Apple 这类高识别度服务单独分组；其余默认回到全局组
- 先清理重复或历史遗留规则，再补最小必要差异，避免规则集越改越碎

### 用户流量经过公网时，用户面服务优先部署在公网云或边缘

- 如果 PVE 位于公司内网、访问依赖 VPN 或回家链路，用户面代理/转发服务默认不要优先放在那里
- 内网 PVE 更适合运维后台、管理面、低频内部任务
- 面向终端用户的入口优先考虑公网云主机、边缘节点或离用户更近的出口

### fork 场景下优先保持部署脚本接近上游，再用 CI 注入自定义变量

- `deploy/.../init.sh` 这类脚本不要长期硬编码仓库 owner、镜像仓库、固定密码或发布参数
- 优先通过环境变量或 CI 中的 shell 替换注入 `REPOSITORY_OWNER`、镜像前缀、版本信息
- 如果既要保留上游可同步性，又要做定制发布，尽量把差异压到 workflow / release 脚本，而不是把核心安装脚本改散

## 注意事项

- 部署问题先看真实流量路径，再看“哪里能装起来”
- 规则治理目标是减少长期维护成本，不是覆盖一切可能域名
- fork 发布链路要优先保证后续同步上游时冲突最小
